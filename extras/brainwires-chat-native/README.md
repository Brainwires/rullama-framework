# brainwires-chat-native

Native chat client for the Brainwires Agent Framework. Targets **iOS, Android, macOS, Windows, Linux, and Web** from a single React Native + TypeScript codebase.

> Sibling project: `extras/brainwires-chat-pwa/` (the original vanilla-JS PWA). This crate is a from-scratch port, not a transpilation.

---

## Architecture

```
extras/brainwires-chat-native/
├── App.tsx, index.js, index.web.tsx     # JS entrypoints (mobile/desktop, web)
├── src/
│   ├── components/ui/                   # Gluestack UI v2 components (CLI-managed)
│   ├── lib/bridge.ts                    # JS surface of the FFI bridge
│   └── screens/                         # App screens
├── android/, ios/                       # generated by RN CLI (this host)
├── macos/                               # added by react-native-macos-init (Mac host)
├── windows/                             # added by react-native-windows-init (Windows host)
├── desktop-linux/src-tauri/             # Tauri 2 wrapper for Linux
├── bridge/                              # Rust FFI crate (uniffi → Kotlin/Swift)
├── vite.config.ts                       # Web bundler (RN-Web)
└── tailwind.config.js, global.css       # NativeWind / Tailwind v3
```

| Target | Runtime | Framework integration |
|--------|---------|------------------------|
| iOS    | React Native (Hermes) | `bridge/` → `xcframework` via uniffi |
| Android | React Native (Hermes) | `bridge/` → `.so` via cargo-ndk + uniffi |
| macOS  | react-native-macos | `bridge/` → static lib via uniffi |
| Windows | react-native-windows | `bridge/` → `.dll` via uniffi (C++/WinRT shim) |
| Linux  | Tauri 2 wrapping the RN-Web build | `bridge/` linked **directly** as a path-dep — Rust-host, no FFI hop |
| Web    | Vite + react-native-web | (UI only — runs against a remote brain server) |

Web and Linux/Tauri share the **same `dist/` artifact** from Vite.

## Toolchain prerequisites

Already on the host (verified Linux dev box): Node ≥ 22, Rust 1.92+, Tauri CLI 2, Java 21.

Install when first targeting a platform:

| For | Install |
|-----|---------|
| Android FFI | `cargo install cargo-ndk` + Android SDK + NDK (`sdkmanager "ndk;26.1.10909125"`) |
| FFI bindings codegen | nothing extra — `bridge/` ships its own `uniffi-bindgen` binary |
| macOS / iOS native build | macOS host with Xcode + CocoaPods (`bundle install` in `ios/`) |
| Windows native build | Windows host with VS 2022 + Win10 SDK |
| Linux Tauri build deps | `apt install libwebkit2gtk-4.1-dev libssl-dev libayatana-appindicator3-dev librsvg2-dev` |

## First-time bring-up (Linux dev host)

```bash
# 1. Install JS deps
npm install

# 2. Initialise Gluestack UI v2 (replaces the minimal stub provider, pulls in components catalog)
npx gluestack-ui@latest init

# 3. Build the Rust FFI bridge once (Linux host = native target)
npm run bridge:build

# 4. Run web target — fast iteration loop
npm run web

# 5. Run Linux/Tauri target (consumes the same Vite build)
npm run linux
```

## Per-target build

### Web (any host)
```bash
npm run web           # vite dev server
npm run web:build     # production bundle to ./dist
npm run web:preview   # preview the prod bundle
```

### Linux (Linux host)
```bash
npm run linux         # cargo tauri dev — auto-runs `npm run web` first
npm run linux:build   # cargo tauri build (set bundle.active=true in tauri.conf.json first)
```

### Android (any host with SDK + NDK + cargo-ndk)
```bash
# 1. Cross-compile bridge to Android targets
cd bridge && cargo ndk -t arm64-v8a -t armeabi-v7a -t x86_64 -o ../android/app/src/main/jniLibs build --release && cd ..

# 2. Generate Kotlin bindings from the freshly-built lib
npm run bridge:bindings:kotlin

# 3. Build / run the app
npm run android
```

### iOS (Mac host with Xcode)
```bash
# 1. Cross-compile bridge to Apple targets and lipo into an xcframework
#    (see ios/Bridge.xcframework section of bridge/README.md)
cd bridge && ./build-xcframework.sh && cd ..

# 2. Generate Swift bindings
npm run bridge:bindings:swift

# 3. Pod install + build
cd ios && bundle install && bundle exec pod install && cd ..
npm run ios
```

### macOS (Mac host with Xcode) — first-time only
The `macos/` project doesn't exist yet. On a Mac:
```bash
npx react-native-macos-init
```
This adds `macos/`, `Podfile.macos`, etc. After that, normal flow:
```bash
cd macos && bundle exec pod install && cd ..
npm run macos
```

### Windows (Windows host with VS 2022) — first-time only
The `windows/` project doesn't exist yet. On Windows:
```powershell
npx react-native-windows-init --overwrite
```
After that:
```powershell
npm run windows
```

## FFI bridge (`bridge/`)

The bridge crate is a workspace member of the parent (root `Cargo.toml`). It's a `cdylib + staticlib + rlib` exposing framework functions to JS via uniffi.

- `bridge/src/lib.rs` — `#[uniffi::export]` annotated functions
- `bridge/src/bin/uniffi-bindgen.rs` — bindings codegen entrypoint
- `bridge/uniffi.toml` — Kotlin/Swift package config

Currently exposes a single smoke-test: `framework_version()`. Add framework crates as feature-gated dependencies in `bridge/Cargo.toml` as the chat app pulls them in.

To regenerate language bindings after editing `lib.rs`:
```bash
npm run bridge:build              # build the cdylib first (bindgen reads the .so/.dylib)
npm run bridge:bindings:kotlin    # → android/app/src/main/java
npm run bridge:bindings:swift     # → ios/generated
```

On Linux/Tauri, the bridge is linked **directly** as a Rust path-dep — no uniffi hop. See `desktop-linux/src-tauri/src/lib.rs`.

## Gluestack UI v2 setup

The scaffold ships a minimal `GluestackUIProvider` stub at `src/components/ui/gluestack-ui-provider/` so the app renders before the official CLI runs. Once you `npm install`, run:

```bash
npx gluestack-ui@latest init
```

This will overwrite the stub provider with the full themed version (with proper `vars` for light/dark/system) and let you pull in components:

```bash
npx gluestack-ui@latest add button input box text
```

Components are copied into `src/components/ui/` (shadcn-style) — fully owned by this project, not imported from a node_module.

## Known gaps

- `bundle.active` is `false` in `desktop-linux/src-tauri/tauri.conf.json` so dev works without icon files. Run `cargo tauri icon ./icon-source.png` from `desktop-linux/src-tauri/` and flip the flag before `cargo tauri build`.
- iOS `xcframework` build script not yet written (see `bridge/build-xcframework.sh` reference above) — write when first targeting iOS.
- macOS / Windows native folders must be initialised on their own hosts (`npx react-native-macos-init`, `npx react-native-windows-init`) — can't init from Linux.

### Notes on web pipeline quirks (already resolved in `vite.config.ts`)

- `nativewind/babel` pulls in `react-native-worklets/plugin` — `react-native-worklets` is in deps for that reason, harmless on web (the plugin no-ops without reanimated calls).
- `vite.config.ts` ships a small inline plugin (`fix-rn-css-interop-jsx`) that pre-transforms `react-native-css-interop/dist/doctor.js` and `nativewind/dist/doctor.js`, both of which ship raw JSX (a runtime pragma-check) that rollup-commonjs can't parse without help.
- The `@vitejs/plugin-react` config sets `babelrc: false, configFile: false` so Vite ignores the project's `babel.config.js` (which carries `react-native-reanimated/plugin` for Metro only).
